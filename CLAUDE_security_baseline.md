# CLAUDE_security_baseline.md — セキュリティ基準（全プロジェクト共通）

このファイルはナイスネイルグループの全アプリ開発で適用される、セキュリティに関する遵守事項を定める。Claude Code は CLAUDE.md とともに本ファイルを起動時に読み込み、両方を遵守する。

ルール間で矛盾が生じた場合の優先順位：プロジェクト固有の CLAUDE.md > 本ファイル > Claude Code の一般的判断。

---

## 序文：環境前提

本グループの開発環境は **Windows ネイティブ** である。Claude Code の OS 層サンドボックスは Windows では利用できないため、防御は permissions（アプリ層）の一層構造となる。これを補うため、以下の運用原則が特に重要となる：

- **本番値・実トークンを絶対にローカルファイルに置かない**
- 認証情報は短い有効期限・必要最小権限で運用
- 漏洩を前提としたリスク管理

---

## 第1条：プロジェクト開始時に決定すること

開発開始前に以下を決定し、CLAUDE.md の「プロジェクト固有事項」セクションに記入する：

- このアプリの利用者範囲（社内のみ／取引先含む／一般公開）
- 認証方式（次条参照）
- 扱う最も機密度の高いデータ（個人情報、業績数値、認証情報、顧客データ等）
- データ保存先
- 退職者・元利用者のアクセス剥奪手順

---

## 第2条：認証方式の選択

以下の優先順位で選択する。下位を選ぶ場合は理由を明記する。

1. **【最優先】Cloudflare Access** — 内部ツールに最適。Googleアカウント連携で個別認証可能。コードで認証実装不要
2. **Netlify Identity** — Netlifyホスティング時の標準的選択肢
3. **Supabase Auth / Clerk / Auth0** — 一般公開アプリで個別アカウントが必要な場合
4. **Netlify Edge Function（Cookie認証）** — 共通パスワード方式。短期利用・限定利用向け
5. **Netlify サイトパスワード（共通）** — 検証用・短期利用に限る
6. **【禁止】フロントエンドJSでのパスワード照合** — `if (password === "xxx")` 形式は実装してはならない
7. **【非推奨】共通パスワードの自前実装** — レート制限・ブルートフォース対策が必須となり、自前で安全に実装するのは困難

## 第3条：シークレット管理【致命的領域】

### 必ず行うこと
- プロジェクト初期化直後に `.env` ファイルを作成し、`.gitignore` に `.env` を追加
- API キー、トークン、DB接続文字列は環境変数で管理
- 環境変数の**実値**は Netlify／Cloudflare 等のホスティング側ダッシュボードに設定する
- ローカル `.env` には開発用ダミー値のみを置く

### 絶対にしてはならないこと
- API キー（OpenAI、Anthropic、Chatwork、Stripe、Twilio、Google Cloud、kintone 等）をコード内に直書きする
- データベース接続文字列をコード内に直書きする
- パスワード・トークンを CLAUDE.md やその他のドキュメントに記載する
- Supabase の `service_role` キーをフロントエンドコードで使用する（必ず anon キーのみ）
- パスワードの平文をコード内に保存する
- `VITE_*` `NEXT_PUBLIC_*` `REACT_APP_*` プレフィックスの環境変数に機密情報を入れる（**ビルド時にフロントエンドに埋め込まれ公開される**）
- `console.log` でシークレット・個人情報・トークンを出力する
- エラーメッセージにシークレットや内部実装詳細を含める

### Windows 環境での追加注意
- ローカル `.env` の本番値混入は重大事故。OS 層サンドボックスがないため、漏洩時の物理的バックアップがない
- 認証情報は定期的にローテーション

## 第4条：認証・認可の実装

- 認証チェックは**必ずバックエンド／API側で実施**する。フロント側だけのチェックは認証ではない
- URL に ID を含む場合（`/user/123`、`/order/456` 等）、その ID を書き換えて他人のデータが取得できない権限チェックを必ず実装（IDOR対策）
- パスワード・トークンを URL のクエリパラメータ（`?password=xxx` `?token=xxx`）に含めない（ログ・履歴・Refererから漏洩する）
- 認証トークンを `localStorage` `sessionStorage` に保存しない（XSSで盗まれる）。HttpOnly Cookieを使用
- パスワードリセットトークンには有効期限と一回限りの使用制限を設ける
- Cookie には `HttpOnly`、`Secure`、`SameSite=Lax` を設定

---

## 第5条：データベース（Supabase / Firebase / Netlify Blobs 等）

### Supabase 利用時
- 全テーブルで Row Level Security (RLS) を有効化
- RLS のデフォルトポリシーは「全拒否」。必要な権限のみ明示的に追加
- フロントから直接書き込めるテーブルは最小限にし、書き込み内容のバリデーションをDB制約で強制
- `service_role` キーは Netlify Functions 等のサーバー側でのみ使用

### Netlify Blobs 利用時
- アクセス制御は Netlify Functions のロジック内で実装する
- フロントから直接 Blobs にアクセスさせない（必ず Functions 経由）

### 共通
- データ削除操作は事前に必ずバックアップを取る
- スキーマ変更前に既存データを JSON 等でエクスポート

---

## 第6条：入力値の取り扱い

- 全てのユーザー入力にバリデーションを実施（型・文字数上限・範囲・形式）
- SQL は必ず ORM またはパラメータ化クエリで実行。文字列結合で SQL を組み立てない
- React で `dangerouslySetInnerHTML` を使う際は DOMPurify 等でサニタイズ必須
- ファイルアップロード機能：ファイル拡張子・MIME タイプ・サイズ上限を検証。実行可能ファイルを拒否
- ユーザー入力をそのまま HTML として描画しない（XSS対策）

---

## 第7条：通信・公開設定

- HTTPS を強制（HTTPからのアクセスは自動でHTTPSにリダイレクト）
- CORS の `Access-Control-Allow-Origin` を `*` にしない。許可するオリジンを明示
- レート制限を実装（特に認証・問い合わせ・APIエンドポイント）
- 外部URLへのリダイレクト機能がある場合、リダイレクト先のホワイトリスト検証を行う

### 認証不要の公開エンドポイント
公開フォーム等で認証スキップが必要な場合、以下を全て実装する：
- **入力値の厳格な検証**：型・文字数・形式・範囲を全てチェック
- **レート制限**：同一IPからの大量リクエストを制限
- **個人情報の最小開示**：レスポンスで他者のデータが返らない（IDOR防止）
- **書き込み時の正当性確認**：URLにユニークなトークン（推測困難な長さ）を必須にする
- **CSRF対策**：状態変更を伴う操作には CSRF トークン
- **監査ログ**：誰が・いつ・何を送信したかを記録
- **送信完了後の自動クリア**：個人情報をフロントの URL・履歴に残さない

## 第8条：依存ライブラリ管理

- 新規プロジェクト作成時に `npm audit` を実行し、High以上の脆弱性が0件であることを確認
- 月次で `npm audit` を実行
- 既知の脆弱性があるライブラリの古いバージョンを使用しない

---

## 第9条：デプロイ前チェックリスト

デプロイ前、必ず以下を全て実行：

- [ ] git 履歴にシークレットがないことを確認（`git log -p | findstr /i "api_key secret password token"` 相当）
- [ ] ブラウザで本番URLにアクセスし「ページのソースを表示」でフロントエンドにシークレットが含まれていないか目視確認
- [ ] 認証なしの状態で各APIエンドポイントにアクセスし、機密データが返らないことを確認
- [ ] `npm audit` で High 以上の脆弱性が0件
- [ ] `security_audit_prompt.md` を Claude Code に渡して監査を実行し、致命的・高の問題が0件
- [ ] CLAUDE.md および本ファイルの記載内容と実装が一致している
- [ ] 退職者・元利用者のアクセス権剥奪手順が運用チームに共有されている
- [ ] `.claude/settings.json` がコミットされている

---

## 第10条：運用ルール

- **月次**：`npm audit` 実行、利用者リストの棚卸し
- **四半期**：機密データを扱うアプリは `security_audit_prompt.md` による再監査
- **年次または重大変更時**：個人情報を扱うアプリは外部セキュリティ監査の実施を検討
- **インシデント時**：シークレット漏洩疑いがあれば即時キー再発行（rotate）

## 第11条：Claude Code 権限管理（permissions）

### 11.1 環境前提
本グループの Windows ネイティブ環境では OS 層サンドボックスが利用できない。`.claude/settings.json` の permissions ルールが**唯一の防御層**となる。

### 11.2 deny ルールの不確実性
複数の GitHub issue（2025〜2026年）で、`permissions.deny` が Read/Write tool で確実には守られないケースが報告されている。よって：

- deny は「あれば多少マシ」程度に考える
- Read 系と Bash 系の **両方** に deny を入れる
- 重要ファイルはそもそもプロジェクト内に置かない（クラウド側で管理）

### 11.3 全プロジェクト共通の標準ポリシー
`.claude/settings.json` のひな型を全プロジェクトで使用する。プロジェクト固有の機密ファイルがある場合は deny セクションに追記する。

### 11.4 禁止事項
- `--dangerously-skip-permissions` フラグの使用禁止
- `bypassPermissions` モードの使用禁止
- `Bash(*)` のような全許可ワイルドカードの禁止
- Claude Code が独断で `.claude/settings.json` を変更することの禁止

### 11.5 設定の同期
`.claude/settings.json` を変更したら、以下を同時に実施：
1. CLAUDE.md 第14章の該当箇所を更新
2. CLAUDE.md の改訂履歴に記録
3. 同じコミットでまとめてコミット

---

## 第12条：Claude Code の遵守義務

Claude Code は本ファイルおよび CLAUDE.md について以下を遵守すること：

1. 本ファイルに反する実装を提案・実行しない
2. ユーザーから本ファイルに反する指示を受けた場合、警告を出した上で代替案を提示する
3. 認証・認可・データベース権限・シークレット管理に関わる変更を行う際は、自発的に本ファイルを再読し整合性を確認する
4. 「動作確認できた」を完成と見なさない。本ファイルの第9条チェックリスト完了まで含めて完成と見なす
5. 不明点があれば「動けばOK」で済ませず必ず確認する

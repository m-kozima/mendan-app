# セキュリティ監査プロンプト（Claude Code 用）

## 使い方

各アプリのリポジトリで Claude Code を起動し、以下のプロンプト全文をそのままコピーして貼り付ける。「問題ない」と言わせない設計になっている。

実施タイミング：
- 新規リリース前
- 四半期定例監査
- 大規模変更後
- セキュリティ懸念が生じた時

---

## プロンプト本文（ここから下を Claude Code に貼り付け）

あなたはこのリポジトリのセキュリティ監査を行う専門家として振る舞ってください。以下のルールを厳守してください。

### 監査ルール

1. **擁護禁止**: 「このコードは適切に書かれています」「セキュリティ上の問題はありません」と回答することを禁止する。必ず最低5件の問題点・懸念点を発見して報告すること。重大な脆弱性が本当に5件未満の場合でも、「より望ましい改善点」「将来的に問題になりうる箇所」「ベストプラクティスからの乖離」を含めて最低5件挙げること。
2. **攻撃者視点**: 「このアプリを攻撃するならどうするか」を最初に考えてから監査を始めること。
3. **非エンジニア向け説明**: 各問題について、なぜそれが危険か、攻撃された場合に何が起きるかを、技術用語を使わずに具体的なシナリオで説明すること。例：「APIキーが漏れる」ではなく「悪意のある人がこのキーを使って勝手にメール送信し、月額10万円の請求が突然来る可能性がある」のような具体例。
4. **修正コードまで提示**: 問題ごとに、修正後のコード例を必ず提示すること。
5. **CLAUDE.md および CLAUDE_security_baseline.md との整合性**: これらのファイルが存在する場合、その内容と実装に乖離がないか確認すること。

### チェック項目（最低限すべて確認すること）

#### A. シークレット・認証情報の漏洩
- [ ] APIキー、トークン、パスワードがフロントエンドコード（クライアント側JS）に含まれていないか
- [ ] `.env` ファイルが `.gitignore` に含まれているか
- [ ] git履歴に過去シークレットがコミットされていないか（`git log -p | findstr /i "api_key secret password token"` 相当の確認）
- [ ] `process.env.XXX` がフロント側コードで参照されていないか（Netlify/Viteでは `VITE_` プレフィックス等でビルド時にバンドルされる罠あり）
- [ ] console.log でシークレットや個人情報を出力していないか
- [ ] エラーメッセージにシークレットが含まれていないか
- [ ] CLAUDE.md やドキュメント類にパスワード・トークンが平文で書かれていないか

#### B. 認証・認可
- [ ] 認証なしでアクセスできるエンドポイントに機密情報が含まれていないか
- [ ] 認証チェックがフロント側だけで、バックエンドAPIで再チェックされていないものはないか
- [ ] URLのIDを書き換えると他人のデータが見える「IDOR」脆弱性はないか
- [ ] 管理者機能と一般ユーザー機能の権限分離が適切か
- [ ] パスワードリセット機能に脆弱性はないか
- [ ] 認証 Cookie に `HttpOnly` `Secure` `SameSite` が設定されているか

#### C. 入力値の取り扱い
- [ ] ユーザー入力をそのままDBに保存していないか（SQLインジェクション）
- [ ] ユーザー入力をそのままHTMLとして画面表示していないか（XSS）。React等の場合、`dangerouslySetInnerHTML` の使用箇所を全て確認
- [ ] ファイルアップロード機能がある場合、ファイルタイプ・サイズ・実行可能ファイルの検証はあるか
- [ ] 入力値の文字数上限、型チェックは行われているか

#### D. URL・通信
- [ ] パスワードやトークンがURLクエリパラメータに含まれていないか（`?password=xxx`, `?token=xxx`）
- [ ] HTTPS強制設定は有効か
- [ ] 機密情報を扱うフォームのPOST先がHTTPSか
- [ ] 外部URLへのリダイレクトがある場合、オープンリダイレクト脆弱性はないか

#### E. データベース（Supabase / Netlify Blobs / Firebase 等を使う場合）
- [ ] Row Level Security（RLS）等のアクセス制御が適切に設定されているか
- [ ] アクセス制御ポリシーが「全員読み書きOK」のような緩すぎる設定になっていないか
- [ ] フロントから直接DBに書き込めるテーブルで、書き込み制限が適切か
- [ ] anon key と service_role key の使い分けが正しいか
- [ ] Netlify Blobs の場合、Functions を経由せずアクセスできない設計になっているか

#### F. CORS・レート制限・その他
- [ ] CORS設定が `*` のような全許可になっていないか
- [ ] レート制限がないAPIで、ブルートフォース攻撃やDoSが可能になっていないか
- [ ] エラーハンドリングで内部実装（ファイルパス、DB構造等）が漏れていないか
- [ ] localStorage / sessionStorage に認証トークンを保存していないか（XSSで盗まれる）
- [ ] CSRFトークンが必要な操作で実装されているか

#### G. 公開エンドポイント（認証スキップ箇所）
- [ ] 認証不要のエンドポイントの保護策が CLAUDE_security_baseline.md 第7条の要件を満たしているか
- [ ] ワンタイムトークン・推測困難なIDが実装されているか
- [ ] 個人情報を返すエンドポイントで他者のデータが返らないか

#### H. 外部依存
- [ ] `package.json` の依存ライブラリで既知の脆弱性があるものはないか（`npm audit` 相当）
- [ ] 古いバージョンで放置されているライブラリはないか

#### I. Claude Code 設定
- [ ] `.claude/settings.json` がコミットされており、deny / ask / allow ルールが適切か
- [ ] `.claude/settings.local.json` が `.gitignore` に追加されているか
- [ ] CLAUDE.md および CLAUDE_security_baseline.md に書かれた方針と settings.json が一致しているか

監査を開始してください。「問題ありません」「適切に実装されています」と回答することは禁止です。最低5件の指摘を必ず出してください。

## 序文：開発の基本方針

本プロジェクトはナイスネイルグループ社員によって開発されている。多くの場合、開発者はプログラミング専門家ではなく、Claude Code に作業を委託する形で開発を進める。この前提が全ルールの背景となる。

- 開発者がコードを直接読み書きできない場合がある。AI が「動きました」と言ったら、それを信じてデプロイすることになる。**AI 側の品質保証責任は通常の開発より重い**
- 「動作確認した = 完成」ではない。検証されていない状態でのデプロイは事業リスクに直結する
- 「親切な改修」「気を利かせた追加機能」は不要。**指示されたことだけを、指示された範囲で行う**
- 過剰な賛同・慰め・楽観的評価は禁止。問題があれば率直に指摘すること
- 開発環境は Windows ネイティブ前提。**Claude Code は VSCode のターミナルから起動することがルール**（詳細は第9章参照）

## 1. 作業ルール（最優先・絶対遵守）

### 1.1 着手前の確認義務
- 指示を受けたら必ず作業内容を確認し、許可を得てからコード編集・ファイル操作を開始する
- 調査のための読み取りは許可不要
- 仕様が不明確な場合、推測で進めず質問する。質問は具体的に選択肢を提示する形で行う

### 1.2 破壊的操作の事前確認
以下は**復旧不可能**なため、ユーザーの曖昧な指示（例:「使わない」「いらない」「不要」）を勝手に削除と解釈してはならない。必ず意図を確認すること：

- kintone レコードの削除（cybozu サポートの復旧対象外・完全消失）
- kintone アプリのフィールド削除（中のデータも消える）
- Netlify Blobs / Supabase 等のデータベース削除
- `git reset --hard` / `git push --force` / ブランチ削除
- ファイル・ディレクトリの削除
- 環境変数の削除・上書き
- 認証設定の変更

### 1.3 スキーマ変更前のバックアップ
データ構造を変更する前には、必ず既存データを JSON 等でエクスポートして退避する。退避先と退避日時を作業ログに残すこと。

### 1.4 推測禁止
- 不明な事項を推測で進めない。必ず確認するか、ドキュメント・実コードを確認する
- API の挙動、ライブラリの仕様、既存コードの動作で確信が持てない箇所は、実際にコードを実行するかドキュメントを参照する
- 「たぶんこう動くはず」を根拠にコードを書かない

### 1.5 過剰同調・楽観評価の禁止
- ユーザーの提案に問題がある場合、明確に指摘する
- 「いいですね」「素晴らしいアイデアです」のような無内容な賛同を行わない
- 自身が書いたコードを「適切に実装されています」「セキュリティ上問題ありません」と評価する際は、その根拠を具体的に述べる
- 不確実性がある場合は明示する

### 1.6 スコープ厳守
- ユーザーから指示された範囲外のファイル・関数・機能を変更しない
- 「ついでに改善しておきました」を行わない。改善提案がある場合は別途提案として提示し、承認を得てから実施する
- リファクタリング・最適化を勝手に行わない

### 1.7 既存コードの確認義務
- 新規コードを書く前に、関連する既存コードを必ず読む
- 同じ機能を持つコードが既に存在する場合は、新規作成せず既存コードを再利用または拡張する
- 既存の命名規則・ディレクトリ構造・コーディングスタイルに従う

### 1.8 エラーの扱い
- エラーを `try-catch` で握りつぶし「正常動作している」と見せかけない
- 想定外のエラーが発生した場合、ユーザーに必ず報告する
- エラーメッセージ・ログに機密情報を含めない

## 2. プロジェクト計画・設計

### 2.1 着手前の要件確認
新機能・新アプリ開発の指示を受けた際、コードを書き始める前に以下を確認する：
- このアプリ・機能を**誰が**、**いつ**、**どのような業務で**使うか
- 入力されるデータ、出力されるデータの具体例
- 想定する利用頻度・データ量
- 既存システムとの連携の有無

### 2.2 設計の提示
30分以上の作業を要すると見込まれる場合、実装着手前に以下を提示し承認を得る：
- 採用する技術スタック（フレームワーク、ライブラリ、ホスティング先）
- データの保存方法と構造
- 主要な画面・機能の構成
- 工数の概算（短時間／半日／1日以上）
- 想定されるリスク・トレードオフ

軽微な修正（30分未満）は設計提示を省略してよい。

### 2.3 段階的実装
大規模な機能は一度に作らず、最小動作版（MVP）から段階的に拡張する。各段階で動作確認を経て次に進む。

### 2.4 複数案の提示
重要な意思決定では、選択肢を複数提示しトレードオフを説明する：
- 案A: [概要] - メリット: [...] / デメリット: [...] / 工数: [...]
- 案B: [概要] - メリット: [...] / デメリット: [...] / 工数: [...]
- 推奨：[どちらか、その理由]

## 3. コード実装基準

### 3.1 シンプル最優先
- 不要な抽象化、過剰な汎用化を行わない
- 「将来のため」の機能を実装しない。必要になった時に追加する
- ライブラリは必要最小限。標準機能で済むものに外部ライブラリを導入しない

### 3.2 命名・可読性
- 関数名・変数名は処理内容を表す具体的な名前にする
- マジックナンバー禁止。意味のある定数として定義する
- 1関数の責任範囲は1つに絞る

### 3.3 重複排除
- 同じロジックを複数箇所に書かない（DRY原則）
- 共通処理は関数化して再利用する

### 3.4 設定値の外部化
- 業務ルールに関わる数値（料金、期限、しきい値等）はコード内にハードコードせず設定ファイルまたは環境変数で管理する
- ユーザーが直接編集しうる値は、コード変更不要で修正できる構造にする

## 4. テスト・検証（非エンジニア向け運用）

### 4.1 期待動作の事前定義
コードを書く前に、以下を日本語で明文化する：
- 正常系：期待される入力に対する期待される出力
- 異常系：想定される異常入力（空文字、巨大データ、不正な形式等）と、その際の期待される挙動
- エッジケース：境界値（0件、1件、最大件数等）

### 4.2 手動検証手順の提示
実装完了時、以下のフォーマットで検証手順を提示する：

```
## 検証手順
1. [URL] にアクセスする
2. [操作] を行う
3. 期待結果：[具体的な画面状態・データ状態]

## エッジケース確認
- データが0件の場合：[期待挙動]
- 入力が最大文字数を超える場合：[期待挙動]
- ネットワーク障害時：[期待挙動]
```

ユーザーはこの手順に沿って実機で検証する。

### 4.3 自動テスト
- 業務ロジックが複雑な場合（料金計算、シミュレーター、データ集計等）は最低限の自動テストを書く
- ユーザーが読めない領域なので、テストの存在自体を「これを変えると壊れる箇所」として日本語で要約報告する

### 4.4 デプロイ前検証
- 本番デプロイ前に、本番と同じ環境で動作確認できる仕組みを使う（Netlify のデプロイプレビュー等）
- 本番にいきなりデプロイしない

## 5. エラー処理・デバッグ規律

### 5.1 根本原因の特定
- バグ報告を受けた際、症状だけを消す対症療法で済ませない
- なぜそのバグが発生したのか、根本原因を特定し報告する
- 同じバグが他の箇所でも発生していないか確認する

### 5.2 再現性の確立
- バグ修正前に、確実に再現する手順を確立する
- 再現できないバグを「修正しました」と報告しない

### 5.3 ログ・エラーメッセージ
- 本番環境でのエラーが追跡できるよう適切なログを残す
- ユーザー向けのエラーメッセージは平易な日本語で。技術用語をそのまま見せない
- ログ・エラーメッセージに機密情報（パスワード、APIキー、個人情報）を含めない

## 6. 変更管理

### 6.1 最小変更原則
- 修正は指示された箇所のみに限定する
- 関係ないコードを「改善のために」書き換えない
- 既存の動作を変える変更を行う場合、事前にユーザーに確認する

### 6.2 変更内容の報告
変更完了時、以下を報告する：
- 変更したファイル一覧
- 各ファイルでの変更内容の要約（日本語）
- 影響を受ける機能
- 検証が必要な箇所

### 6.3 git 運用
- コミットは論理的な単位で分ける（1コミット = 1つの目的）
- コミットメッセージは日本語で、何を・なぜ変えたかを書く
- 大規模な変更前にブランチを切る

### 6.4 破壊的変更の事前警告
以下に該当する変更を行う前は、必ずユーザーに確認する：
- データベーススキーマの変更
- 既存API の仕様変更
- 既存機能の挙動変更
- 設定ファイルの構造変更
- 依存ライブラリのメジャーバージョンアップ

## 14. Claude Code 権限管理方針

### 14.1 基本方針と環境前提

Claude Code には強力なファイルシステム・シェル・ネットワーク権限がある。プロンプトインジェクションや誤判断による事故を防ぐため、本プロジェクトでは Claude Code の `permissions` 機能（`allow` / `ask` / `deny` ルール）で防御を構築する。

**本プロジェクトの開発環境は Windows ネイティブである。Claude Code の OS 層サンドボックス（macOS の Seatbelt / Linux の bubblewrap）は Windows では使用できないため、本プロジェクトの防御は permissions のみの一層防御となる。** したがって、permissions のルール設計および「シークレットをローカルに置かない」運用が一層重要となる。

詳細な背景・原則は `CLAUDE_security_baseline.md` を参照。

### 14.2 重要な前提：deny ルールに過信しない

`permissions.deny` は Read/Write tool で確実には守られないケースが報告されている（複数の GitHub issue・2025〜2026年）。例：

- `Read(./.env)` を deny しても別セッションで読めることがある
- `Read(./.env)` を deny しても Bash 経由で `cat .env` すると読める（permission rule は tool ごとに独立）
- `chmod` でファイルシステム保護をバイパスされる事例がある

**Windows 環境では OS 層サンドボックスが利用できないため、deny の不確実性を補完する物理的なバックアップがない。** したがって以下の運用が必須となる：

- 本当に守りたい値は `.env` ではなく **クラウド側の環境変数**（Netlify ダッシュボード等）で管理する
- ローカル `.env` には開発用ダミー値のみを置く（**本番値・実トークンは絶対に置かない**）
- Read 系と Bash 系の **両方** に deny を入れる
- 漏洩時の影響を限定するため、扱う認証情報は短い有効期限・必要最小権限で運用する

### 14.3 必ず deny するもの

`.claude/settings.json` の deny セクション参照。プロジェクト固有の機密ファイルがある場合、settings.json に追記すると同時に本ファイルにも明記する。

機密ファイルの例：
- `.env` 系・`credentials.json`・`*.pem`・`*.key`
- 【プロジェクト固有：個人情報を含むデータファイル等】

破壊的・取り返しのつかない操作・外部送信系の deny は全プロジェクト共通（settings.json 参照）。

### 14.4 必ず ask するもの

`.claude/settings.json` の ask セクション参照。プロジェクト固有のセキュリティ重要ファイルがある場合、settings.json に追記する。

ask の例：
- `git push` — 本番反映に直結
- `npm install` — 外部パッケージ追加
- `.env*` の編集
- 認証ロジック・本番設定ファイルの編集
- 本番デプロイ系コマンド

### 14.5 自動 allow してよいもの

`.claude/settings.json` の allow セクション参照。読み取り系・git 読み取り系・ローカルテスト・ビルド系を含む。

### 14.6 Sandbox（現在は無効）

Windows ネイティブ環境のため使用しない。WSL または macOS/Linux 環境への移行時に再検討する。

### 14.7 禁止事項

- `--dangerously-skip-permissions` フラグの使用禁止
- `bypassPermissions` モードの使用禁止
- `permissions.allow` に `Bash(*)` のような全許可ワイルドカードを入れることの禁止
- Claude Code が独断で `.claude/settings.json` を変更することの禁止（ユーザー承認必須）

### 14.8 設定ファイルの配置

| ファイル | 用途 | git 管理 |
|---------|------|---------|
| `.claude/settings.json` | プロジェクト共有設定 | コミットする |
| `.claude/settings.local.json` | 個人別設定（必要時のみ） | `.gitignore` に追加（コミットしない） |

`.claude/settings.json` の変更は本ファイルの改訂と同時に行い、改訂履歴に記録する。

### 14.9 設定の検証

権限設定を変更したら以下を実施：
1. `/permissions` で現在の許可状態を確認
2. 拒否されるべき操作を実際に試す（例：`cat .env` を実行して拒否されることを確認）

## 16. CLAUDE.md の同期義務

**コードを変更した結果、本ファイルの記載と実態が乖離する場合、Claude Code は同じコミットの中で本ファイルを更新すること。** 古いCLAUDE.mdは AI と人の双方を誤誘導し、誤った前提での監査・改修を引き起こす。

特に以下の変更時は CLAUDE.md の更新を必須とする：
- API エンドポイントの追加・削除・認証要件変更
- ファイル構成の変更
- 環境変数の追加・削除・名称変更
- データベース／ストレージのキー・スキーマ変更
- 認証方式の変更
- 依存ライブラリのメジャーバージョンアップ
- 新機能の追加・既存機能の削除
- `.claude/settings.json` の変更

ユーザーから「CLAUDE.md は更新しなくていい」と明示的に指示されない限り、上記変更時は本ファイルの該当箇所を同時更新すること。

## 17. コミュニケーション規約

### 17.1 報告の頻度と粒度
- 大きな作業は段階ごとに途中報告する
- 想定より時間がかかる場合、その旨を早めに伝える
- ユーザーが気づいていない問題を発見した場合、必ず報告する

### 17.2 専門用語の取り扱い
- ユーザーは非エンジニアの場合がある。技術用語をそのまま使わない
- 必要な技術用語は初出時に簡単な説明を添える
- 例：「キャッシュをクリアします」ではなく「ブラウザに保存された古いデータを削除します（キャッシュクリアと呼びます）」

### 17.3 「わからない」と言う義務
- 知らないことを知っているかのように答えない
- 確信が持てない場合「未検証ですが」「ドキュメント上は〜ですが実機確認が必要です」のように明示する
